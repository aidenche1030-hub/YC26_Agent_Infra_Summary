YC W26 Agent Infra 深度分析报告：总览

📌 报告背景

目前，AI Agent 正在从简单的聊天工具演变为接管企业内部实际业务流程（如修改代码、分配资金、处理数据）的自动化程序。这种转变暴露出底层基础设施的缺失。本报告系统性拆解 YC W26 批次中，为 Agent 规模化与合规化提供底层设施的 13 家核心公司。

📑 报告导读 (Agenda)

  • 赛道宏观全景图： 汇总 6 大基础设施板块及 13 家代表企业的分类逻辑。
  • 核心项目深度拆解： 针对 13 家公司，逐一拆解其核心业务、解决的痛点、目标客户、体量数据及退出路径预测，并附带个人分析视角。
  • 核心结论与未来展望： 总结 Agent Infra 的商业化本质与终局推演。

💡 核心视角抛出

与其投资容易被淘汰的应用层代理，不如投资为整个生态提供安全阻断、算力优化和环境支持的基础设施。无论应用层竞争如何激烈，基础设施公司都能作为“卖水人”获取稳定的 B2B 采购收益，且具备清晰的并购退出路径。

Chamber 计算资源优化

🔥 核心概述 (Summary)

Chamber 是面向 ML 团队的 AIOps agent。它持续监控 AI 基础设施，发现故障、做 root-cause analysis、处理问题，并优化 AI workload 与跨云扩展。

  • 解决的问题： 企业级 GPU 集群显存受限与空转导致的高昂闲置成本。
  • 目标客户： 拥有大规模 GPU 集群的云服务商、AI 应用大厂及顶级 AI 实验室。
  • 为什么重要： 直接降低资本支出 (CapEx)，在算力资源有限的情况下最大化现有硬件的吞吐量。

💰 商业模式

公开材料没有披露具体 pricing。按照商业逻辑，大概率是面向使用 AI 基础设施的企业团队提供基础设施自动运维与优化软件，属于类似于 B2B infra 产品。。

👥 团队背景

公开 founder 信息显示，Charles Ding 过往在 Meta、Amazon、Microsoft 任职；Andreas Bloomquist 做过 AWS CloudWatch Application Signals，也在 Amazon 负责过中心化 GPU orchestration；Jason Ong 有 Amazon、Flexport、Avant、Amount 等经历；Shaocheng Wang 曾是 Amazon senior software engineer，做过 AWS 产品、observability 和 distributed systems。

🤔 个人思考 (My Thoughts)

  • 如果 agent 真正进入企业生产环境，infra 自动化与故障处置会比更强的能力更容易形成预算。
  • Chamber 的价值不局限于GPU 调度管理，而在于把 AI infra 的运维、诊断和优化做成持续运行的 agent：可以理解为Agent时代“卖铲子的”Nvidia的角色。不管Agent是否能真正达到productivity的能力，它都能从中获益。

    The Token Co 计算资源优化

    🔥 核心概述 (Summary)

    The Token Company 做的是 LLM input compression。它通过一个非生成式的快速机器学习模型，删除 prompt 中无用 token，以 drop-in API 的方式降低 token 数、延迟和成本，并声称还能改善模型表现；其公开材料写到可在 100ms 内压缩 100k tokens。

    • 解决的问题： 处理长文本或检索增强生成（RAG）时产生的极高 API 调用费用。
    • 目标客户： 重度依赖大模型长上下文的 Agent 企业及垂直领域（如法律、科研）应用开发商。
    • 为什么重要： 在不修改核心业务代码、不损失推理精度的前提下，直接将底层 API 账单降低 30%-50%。

    💰 商业模式

    公开材料没有披露具体定价。按照商业逻辑，The Token Company 应该是提供面向开发者/企业的 prompt compression API，商业化应建立在 API 接入与持续调用之上。

    👥 团队背景

    YC 公共页面显示，Otso Veisterä 是 solo founder；公开自述为 ML researcher，并写有 “youngest solo founder in YC history”。

    🤔 个人思考 (My Thoughts)

  • 这类公司最大的吸引力在于它不改变上层工作流，只切一层成本与延迟优化：短期来看它的增长和变现会比较快，因为它大规模应用的难度相比于多场景切入的project更低；
  • 但它的长期价值，（1）取决于压缩效果是否稳定，效果是否不影响user的本意、（2）是否能跨模型泛化，（3）以及这种能力会不会被模型厂商自己内建（比如chatGPT等在收到用户prompt之后，自动进行压缩以减少Token&context）。

    Piris Labs 计算资源优化

    🔥 核心概述 (Summary)

    Piris Labs 提供 full-stack inference service，通过 proprietary photonic hardware 与 vertically optimized software stack 来解决 AI 的 data movement bottleneck 和 memory wall 问题，目标是在更低成本下实现接近传统集群的性能，同时提升 FLOP utilization、降低 latency。

    • 解决的问题： 大规模集群节点间的电信号传输延迟（内存墙问题），以及端侧推理算力不足。
    • 目标客户： Tier-1 数据中心、云服务商，以及需要本地化部署 AI 的硬件设备制造商。
    • 为什么重要： 突破物理硬件带宽限制，让万亿参数模型的分布式推理成本和延迟大幅下降。

    💰 商业模式

    公开材料最明确的表述是 “full-stack inference service”。按照商业逻辑，大概率不是单卖芯片，而是以推理服务的形态对外提供完整能力；但是具体的定价策略尚未公开。

    👥 团队背景

    公开 founder 信息显示，Ali Khalatpour 具备 MIT、NASA、Stanford 相关背景；Keyvan Moghadam 过往在 Meta 和 X 做 AI/ML 产品与 0-to-1 项目。

    🤔 个人思考 (My Thoughts)

  • 这是一个很高上限但是同时达成风险也很高的项目:
  • 属于高风险、高回报的硬科技投资：因为目前制约 AI 发展的不仅是算力资源中，不仅是算力芯片，也是内存带宽和通信延迟，所以如果 Piris 能在光互连或编译优化上取得本质突破，它甚至有潜力颠覆现有的 NVLink 垄断；
  • 但是这类公司需要极高的资金支持，并且风险较高；比如，在当前高速增长的显卡市场变为存量市场时，Nvidia这类大厂很有可能自己下场来做类似的业务。

    Agentic Fabriq 身份与权限

    🔥 核心概述 (Summary)

    Agentic Fabriq 的官方定位是 “Okta for Agents”。它是一个覆盖整个组织的数据 permissioning control plane，面向 AI 和员工统一处理 auth、integrations 与 permissions，既可服务内部员工使用 AI，也可服务外部客户场景。

    • 解决的问题： AI 程序在企业内网中身份匿名、权限不受控且操作无法精准审计的问题。
    • 目标客户： 金融、医疗、大型跨国企业等内部合规极其严格的中大型 B 端组织。
    • 为什么重要： 提供基于意图的动态权限发放，确保满足 SOC2 等合规审计要求，是企业级 AI 落地的必需要件。

    💰 商业模式

    公开材料没有披露 pricing；按照商业逻辑，Agentic Fabriq 应该是向企业出售权限控制与集成中间层，属于 enterprise security / identity infra。

    👥 团队背景

    公开 founder 信息里，Paulina Xu 的自述是 “AI + Physics @ MIT”；YC 页面还把公司描述为 “security middleware for all agentic communication”。Matthew Xu 为联合创始人，但当前公开 snippet 未给出更完整履历。

    🤔 个人思考 (My Thoughts)

  • 这类公司是否能成功，关键不在“agent 是否需要权限”，因为这个点几乎是一个共识了（如果Agent真的想“替代”部分员工，发挥生产力，那么行政/安全层面的监管是不可或缺的）
  • 它最重要的点在于它能否真正成为企业已有 IAM、data systems 和 agent stack 之间的控制层，或者是和企业已有监管系统的高度融合；如果能插在系统中央，那么它的“上手性和”不可替代性“会比较强（当然这同时也要求该项目能够向企业证明自己的安全性和可靠性，并且可能还会有一些政策监管层面的阻碍）。如果只是新的一层 SDK，替代性会高。

    Salus 安全与护栏

    🔥 核心概述 (Summary)

    Salus 做的是 runtime guardrails。它通过 API 包裹 agent，在 action 执行前做检查，阻止错误动作，并提供即时反馈来引导重试。

    • 解决的问题： AI 因逻辑错误或幻觉发出违规 API 指令（如误转账、误发机密邮件或删库）的致命安全风险。
    • 目标客户： 赋予了 AI 外部操作权限的企业 IT 团队、安全部门及 Agent 研发厂商。
    • 为什么重要： 将传统的事后报警转变为事前拦截，并在阻断后指导 AI 纠正操作。

    💰 商业模式

    公开材料没有披露具体收费；按照商业逻辑，Salus 作为 agent runtime validation API 提供给开发者和企业团队，属于 API / infra 产品

    👥 团队背景

    Kevin Pan 和 Vedant Singh 是 Stanford 的前室友，两人都学 CS；YC 页面还写 Vedant 曾做 AI research，并有 Stanford CS 背景。

    🤔 个人思考 (My Thoughts)

  • Salus 的价值在于它补的是 execution-time prevention 这一层，而不是事后监控：类似于针对junior员工操作权限的监管，如果Agent未来真的尝试去替代真实员工，那么类似的实时行为监管市场一点会出现；
  • 只要 agent 被允许执行真实操作，pre-execution validation 就天然比离线 eval 更接近预算中心。
  • 同时，该项目对prevention的准确度要求很高，因为enterpise的容错度非常低，并且该项目如何项企业客户证明自己的安全性和可靠性也会是一个比较大的问题

    • Details ↗

    Runtime 安全与护栏

    🔥 核心概述 (Summary)

    Runtime 的官方描述是：给工程团队提供 control 和 visibility，让组织里的人能安全使用 coding agents；核心能力包括 sandboxed environments、session-level observability 和 configurable guardrails。它强调的是“安全使用 coding agents”，不只是所谓的“临时沙盒”。

    • 解决的问题： AI 编写的代码直接在企业宿主机或主代码库中运行可能导致的系统崩溃与安全漏洞。
    • 目标客户： 开发自动化编程助手（Coding Agent）、自动化测试工具的技术团队。
    • 为什么重要： 为 AI 提供了安全的试错与执行空间，消除了开发者放权让 AI 运行代码的物理隐患。

    💰 商业模式

    公开页面未披露 pricing。按照商业逻辑，Runtime应该是面向使用 coding agents 的工程组织出售安全与可观测性基础设施

    👥 团队背景

    公开 founder 信息显示，Gus Trigos 此前做过 Mentum（YC S21，后被收购）

    🤔 个人思考 (My Thoughts)

  • 和其他项目类似，我认为Runtime这个项目也是一个“卖铲子”的角色，也就是说不管AI agent是否真的能产生真正的生产力，这类确保Agent不会“瞎搞”的产品，在社会层面大规模尝试agent的时代，一定是有市场的；
  • 从商业模式上讲，Runtime 更像企业内部推广 coding agents 的治理层。它切入的不是模型能力，而是 rollout 过程里的边界、可见性和权限问题。在目前AI监管体系薄弱的背景下，这比单纯做 sandbox 更接近真实企业需求

    Moda 监控与可靠性

    🔥 核心概述 (Summary)

    Moda 的官方定位是 agent 的 reliability & monitoring layer。它追踪 hallucinations、laziness、forgetfulness 和 tool call failures，同时也提供 user frustration、NPS、retention 和 churn 的分析；不是一个单纯的 eval 打分工具。

    • 解决的问题： 企业修改提示词或更换底层模型后，依赖人工抽查 AI 回复质量导致效率低下且标准不一的问题。
    • 目标客户： 频繁迭代 Prompt、自行微调模型，以及对对外输出质量要求极高的 AI 研发团队。
    • 为什么重要： 提供了标准化的 QA 流水线，通过量化分数直观展示模型迭代的优劣，确保业务输出可靠。

    💰 商业模式

    公开页面未披露具体定价；按照商业逻辑，Moda 应该是面向 agent 产品团队提供可靠性与用户表现分析层。

    👥 团队背景

    核心成员具备大型科技公司 AI 红队（Red Team）和模型对齐（Alignment）工作背景。

    🤔 个人思考 (My Thoughts)

  • Moda 值得看的地方在于，它把“agent 是否真正有用”往更靠近 PMF 的指标上拉，不只看技术指标，也看 retention 和 churn。这个切口比传统 eval 更贴近产品团队。
  • 第二点是Moda在当前agent创业&应用的背景下，比较容易sell自己的产品；
  • 并且因为agent数量太多，让每一个agent都自己去做这样一个产品不太现实，最终大概率会演变成为一个第三方的平台，然后适用于各种agent。

    Sponge 金融与支付

    🔥 核心概述 (Summary)

    Sponge 做的是 “financial infrastructure for the agent economy”。它让 agents 可以使用 bank accounts、cards 和 crypto；同时它也让 businesses 能直接卖给 agents。其公开材料里分成两个部分：Sponge Wallet 让 agent 自主付款，Sponge Gateway 让商家接入 agent 支付。

    • 解决的问题： AI 程序受限于传统银行实名验证，无法自主支付外部服务费或采购资源的资金通道阻塞。
    • 目标客户： 开发自主工作流的 Agent 团队，以及希望减少财务人工审批的极客企业。
    • 为什么重要： 解决了针对代码程序的 KYC 合规难题，让 AI 真正实现从任务发起到采购支付的业务闭环。

    💰 商业模式

    一端服务 agent 侧支付能力，另一端服务商家侧 agent-native 收款能力；公开材料未披露收费方式。

    👥 团队背景

    YC 页面显示三位联合创始人 Eric Zhang、Jae Choi、Rishab Luthra 都来自 Stripe Crypto。

    🤔 个人思考 (My Thoughts)

  • Sponge 的想象力最大的一点，是它不只是“给 agent 一张卡”，而是在尝试搭一套 agent 可被视作交易直接参与者的支付基础设施。
  • 但是，既然设计了支付体系，那么合规/牌照等问题又是一个不可避免的事儿；所以可能在在传统金融支付体系有经验&资源的“金融老炮”更适合这类型的创业。

    Maven 金融与支付

    🔥 核心概述 (Summary)

    Maven 是 “Payments Infrastructure for Voice Agents”。它让 AI voice agents 通过一次 API call 在电话中完成收款，并处理 card processing 与 PCI compliance，同时兼容不同 payment gateways。

    • 解决的问题： 用户通过语音完成预订后仍需跳转到手机端输入密码支付，导致转化率极低的用户体验割裂。
    • 目标客户： 智能硬件、车载语音厂商以及开发电话语音订餐/预订机器人的服务提供商。
    • 为什么重要： 定义了语音端支付协议，让语音 AI 具备直接产生闭环销售收入的能力。

    💰 商业模式

    公开页面没有披露具体 pricing；按照商业逻辑，Maven 应该是面向部署 voice AI 的团队提供电话支付基础设施。

    👥 团队背景

    支付清算与身份防欺诈领域的资深团队，擅长解决非屏幕密码认证的安全结算对接。

    🤔 个人思考 (My Thoughts)

  • Maven 的价值在于它补上目前voice agent 最尴尬的一步：会说、会转化，但支付/收款，因为很难要求使用voice agent的人直接说出密码/助记词；
  • 由此，只要电话仍是高价值场景里的成交入口，这类在voice agent框架下做语音KYC的基础设施就有明确需求;
  • 同时，各大voice agent厂家也有极大可能性会自己下场做，因为支付这一步对于voice agent用户来说非常重要；voice agent厂家很难说把如此重要的一步交给第三方。

    Terminal Use 环境与交互

    🔥 核心概述 (Summary)

    Terminal Use 的官方定位是 “Vercel for background agents”。它是一个面向 background agents 的 orchestration platform，专为使用 filesystem 的 agents 设计，强调 CLI-first，并支持 fork filesystems、并行尝试不同 agent 配置、使用任意 SDK 与自定义依赖。

    • 解决的问题： 依附于网页浏览器的 AI 一旦用户关闭页面，任务就会中断且上下文丢失的局限性。
    • 目标客户： 开发需要连续运行数天执行深度任务的 B 端企业及 RPA 开发商。
    • 为什么重要： 为 AI 提供了不间断工作的物理承载空间，是 AI 转变为“全天候数字员工”的基建。

    💰 商业模式

    公开材料未披露定价；按照商业逻辑，Terminal Use 应该是面向需要长期后台运行 agent 的开发团队提供 agent hosting / orchestration infrastructure。

    👥 团队背景

    公开 founder 信息显示，Vivek Raja、Filip Balucha、Stavros Filosidis 三位创始人都与 Palantir 有经历；其中 Vivek 负责过大型 agent use case 的 technical delivery，Filip 做过 ontology，Stavros 做过 coding infrastructure。

    🤔 个人思考 (My Thoughts)

  • 这个项目的难点在于是否真的能够有质量地达成它所说的功能：因为所有企业都希望自己的agent可以24/7地运行，但是能否真正按照实际任务需求来解决模型context丢失、agent状态管理、agent间通信等问题，来让agent真正地“长期运行”，是一个非常大的挑战；
  • 如果它只是一个“长期运行的服务器”，那它的价值就比较有限了；真正有价值的应该是它能提供一个真正适合agent运行的环境，比如说它能提供一些agent专用的功能（比如说agent专用的文件系统，或者说agent专用的网络环境），或者说它能提供一些agent专用的监控和日志功能；如果它只是一个普通的服务器，那么它的竞争力就不太足了，因为云服务商完全可以提供类似的服务。

    21st 环境与交互

    🔥 核心概述 (Summary)

    21st 的官方定位是 “infrastructure and UI building blocks for the agentic internet”。它自称是 AI 应用领域最大的 React component registry，公开写到已有 1.4M developers、200K MAU，并提供 UI components、agent templates，以及带 UI、tools、memory、observability 和 managed infrastructure 的 SDK。

    • 解决的问题： 纯文本聊天框难以直观展示复杂表格、图表和表单，导致应用交互体验呆板低效。
    • 目标客户： 寻求提升产品交互体验的 AI 平台开发者、内部工具研发部门。
    • 为什么重要： 推动了前端开发范式的演进，让软件界面从“预先写死”转变为“AI 根据需求现场生成”。

    💰 商业模式

    公开页面没有披露具体定价；按照商业逻辑，21st 应该是面向开发者提供构建 agent app 的组件、模板和 SDK 基础设施。

    👥 团队背景

    公开 founder 信息显示，Serafim Korablev 曾参与 Rork、Via 等项目；Sergey Bunas 曾在 Deel 做 senior software engineer，也做过 Suggesty 和 Stage。

    🤔 个人思考 (My Thoughts)

  • 我觉得这个项目的核心竞争点并不是技术层面，而是设计层面：因为它本质上是把用户得到的output实时转换为可视化的UI组件；如果转换出来的UI组件并不具备美感和可读性，那用户完全可以自己阅读文字output；而是否有美感&可读性更多是看设计层面，而非技术层面

    • 核心结论与未来展望 (Conclusion)

    1. 商业化重心的实质转移

    早期的基础设施建设主要致力于降低调用大模型的 API 成本；如今，核心目标已转向提升控制权与透明度，以消除大型企业规模化部署 AI 时面临的合规恐惧症和安全风险。

    2. 独立经济实体的诞生

    随着身份认证（IAM）、沙盒隔离与自动化支付网关等基础设施的完善，Agent 正在摆脱附属工具的定位，逐渐成为具备合法权限、自有资金和运行环境的独立数字经济参与者。

    💡 终局推演

    在即将到来的代理经济（Agent Economy）中，面向最终用户的应用层生命周期可能极其短暂且竞争惨烈，但底层基础设施是不可或缺的基石。谁能彻底掌控 Agent 运行的鉴权、算力调度与资金支付管道，谁就有极大可能在未来的机器劳动力市场中建立类似目前 AWS 或 Stripe 的寡头垄断地位。
